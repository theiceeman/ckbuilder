# Activity Log - Week 8

## Summary
Week 8 focused on understanding cycle optimization for JavaScript-based CKB scripts. While JavaScript provides superior developer experience and fast iteration compared to native languages like C/Rust, it traditionally consumes significantly more cycles. This week explored three practical optimization techniques that collectively reduced cycle consumption from ~20 million cycles to ~4 million cycles (75% reduction), making JavaScript viability for production CKB scripts.

## Learning & Research

### The Cycle Consumption Problem
- **Baseline Reality**: JavaScript scripts consume ~20 million cycles vs ~5,000 cycles for equivalent C implementation
- **Production Impact**: More cycles = higher transaction fees, making JavaScript seemingly impractical
- **Strategic Question**: Given optimized algorithms and implementations, are there other cycle reduction techniques without rewriting to native languages?
- **Investigation Scope**: Focus on JavaScript-only optimizations while preserving developer experience advantages

### Architecture: Script Evaluation Pipeline
- Studied how CKB scripts execute on CKB-VM (RISC-V runtime)
- CKB-VM provides:
  - 33 registers (32 general purpose + PC register)
  - Fixed 4MB memory segment per execution
  - Memory automatically destroyed after execution completes
- Key insight: Traditional cleanup/resource management is wasteful on CKB-VM

## Optimization Technique 1: Skip Initialization

### Problem
- Dynamic language implementations (like Duktape) require initialization phase
- VM setup: create runtime environment, initialize data structures, configure execution context
- This initialization runs in every single script execution

### Solution
- Use debugger's dump syscall to serialize complete VM state
- Capture all registers and memory state after initialization completes
- Generate executable binary containing pre-initialized state
- Later executions restore state instead of re-initializing

### Implementation Steps
1. Run original duktape with dump syscall enabled
2. Debugger serializes VM state into binary file (dump0.bin)
3. Deploy dump0.bin as the script instead of original duktape
4. New executions restore state from binary, skipping initialization

### Performance Impact
- **Baseline (full init)**: ~20,199,104 total cycles
- **With dump (skipped init)**: ~16,249,542 total cycles
- **Savings**: ~3,949,562 cycles (~19% reduction)

### Key Insight
- Initialization cost is substantial but one-time per script
- Trading small offline cost for repeated online savings across all executions

## Optimization Technique 2: Bytecode Over Source

### Problem
- Previous approach executed JavaScript source directly
- Duktape parses source → bytecode → executes at runtime
- Parsing happens in every single script invocation
- Parsing cycles are pure overhead (same source, same result every time)

### Solution
- Pre-compile JavaScript source to Duktape bytecode format offline
- Ship bytecode binary instead of source to CKB
- Runtime skips parsing phase entirely

### Implementation Steps
1. Use dump_bytecode tool: converts source to bytecode format
2. Pass bytecode file (script.bin) to dump_load0
3. Dump_load0 detects bytecode format (runtime check) and loads accordingly
4. Deploy dumped VM with bytecode executed

### Performance Metrics
- **With bytecode dump (no prior optimization)**: ~9,239,414 cycles during dump
- **Executing from bytecode dump**: ~5,289,852 cycles
- **Combined with Technique 1**: ~5,289,852 cycles
- **Total savings vs baseline**: ~14,909,252 cycles (~75% reduction)

### Tradeoffs
- **Advantage**: Significant cycle savings (parsing eliminated)
- **Disadvantage**: Bytecode file size larger than source
  - Original JavaScript source: ~3.5KB
  - Generated bytecode: ~7.8KB (~2.2x larger)
- **Version Lock**: Bytecode format is NOT guaranteed compatible across Duktape versions
  - Mitigation: Ship Duktape binary with script, lock version in deployment
  - This version locking is feasible because Duktape itself is a smart contract dependency

### Decision Framework
- **Bytecode choice if**: Cycle efficiency > file size efficiency
- **Source choice if**: Smaller deployment footprint > execution cost
- **Recommendation**: Bytecode for long-running or high-frequency scripts

## Optimization Technique 3: Skip Cleanup Work

### Problem
- Traditional programming: cleanup resources before exiting (close files, free memory, etc.)
- CKB-VM provides 4MB memory segment per execution
- Memory segment is completely discarded after script finishes

### The Insight
- Cleanup steps are wasted effort in CKB-VM environment
- As long as script returns correct exit code, memory cleanup is irrelevant
- Entire 4MB segment destroyed anyway — cleanup cycles are pure waste

### Solution
- Provide dump_load0_nocleanup variant
- Skips all resource cleanup after script execution
- Returns result immediately

### Performance Impact
- **With bytecode + cleanup**: ~5,289,852 cycles
- **With bytecode + NO cleanup**: ~3,903,352 cycles
- **Savings from skipping cleanup**: ~1,386,500 cycles (~26% additional reduction)
- **Total combined savings**: From 20M → 3.9M cycles (80% reduction overall)

### Strategic Implications
- Requires mindset shift: CKB-VM is different from traditional environments
- Clean code practices don't always translate to CKB scripts
- Focus on correctness of result, not resource tidiness

## Cumulative Optimization Results

|-------|-----------|--------------|------------|-------------------|
| Baseline | Plain JavaScript | 20,199,104 | — | — |

### Analysis
- JavaScript → 3.9M cycles is 700x more than C (5.5K cycles)
- JavaScript → 3.9M cycles is now viable for many production scenarios
- Cost-benefit: Developer productivity gains justify remaining cycle overhead
- Optimization reaches diminishing returns; C rewrite only saves ~1.4M more cycles

## Key Technical Insights

| Concept | Key Point |
|---------|-----------|
| VM Initialization Cost | Substantial overhead, can be pre-computed and serialized |
| State Serialization | Debugger dumps registers + memory for later restoration |
| Parse vs Execute | Parsing happens per invocation; bytecode skips this |
| Bytecode Tradeoff | File size increase (~2x) for cycle savings (~26%) |
| Cleanup Irrelevance | CKB-VM discards all memory; cleanup cycles are wasted |
| Environment Awareness | Script optimization requires understanding platform constraints |
| Diminishing Returns | Each technique yields smaller gains; combined effect is powerful |
